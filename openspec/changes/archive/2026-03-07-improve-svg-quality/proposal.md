# Change: Improve SVG Generation Quality (improve-svg-quality)

## Why

現在のアーキテクチャでは、LLM（GPT-4o等）がSVGコードを単一ショットでテキストとして直接出力している。この方式では、LLMが精密な座標データ（ベジェ制御点等）を正確に出力する能力に限界があり、出力品質は「子供の絵」レベルにとどまっている。

画像生成AI（DALL-E、Stable Diffusion等）は著作権リスクの観点から業務成果物には使用できないため、LLMテキスト生成の枠内で品質を最大化する必要がある。

対象スコープはダイアグラム、アイコン、インフォグラフィック、フラットデザインイラストまでとし、有機的イラスト（キャラクター、風景等）は対象外とする。

## What Changes

4つの戦略を段階的に導入する：

### 戦略1: コード生成アプローチ（LLM → Pythonコード → SVG）

- LLMにSVGを直接書かせず、SVG生成用のPythonコードを出力させる
- Pythonコード内で数学関数・ループ・ライブラリを活用し、精密なSVGを生成
- サンドボックス環境でコードを実行し、結果のSVGを取得
- **新規依存**: `drawsvg` (MIT license)

### 戦略2: 用途特化プロンプト戦略

- ユーザーの入力を「ダイアグラム」「アイコン」「インフォグラフィック」「フラットイラスト」に分類
- 各用途に最適化された専用システムプロンプトを適用
- カラーパレット、フォント、レイアウトルールを用途別に定義
- LLMによる自動分類 + ユーザー手動選択の両方を提供

### 戦略3: ビジョンフィードバックループ

- LLM生成SVGをレンダリング → GPT-4V（ビジョンモデル）が評価 → 改善指示 → LLM修正
- 画像「生成」ではなく画像「認識/評価」のため著作権リスクなし
- 最大3-5回の反復で品質を漸進的に改善
- ユーザーがオプトイン/アウトで制御可能

### 戦略4: コンポーネントライブラリ + LLMコンポジション（課題特化設計）

戦略1〜3の実装後も残存する2つの品質課題を**構造的に解決**するための設計:

- **課題1: 人・顔の描画品質** — プロ品質のSVG人物パーツ（`people/` カテゴリ）を事前にキュレーションし、LLMは配置のみを指示。LLMがトークン空間で有機的な形状を生成する問題を根本回避
- **課題2: ダイアグラムの結線ずれ** — 接続ポート（anchors）付きコンポーネント + コンポジションエンジンによるマンハッタンルーティング自動結線。LLMは座標を一切指定せず、アンカー名で接続先を指定
- MIT/CC0ライセンスのSVGパーツライブラリを内蔵（`people/`, `shapes/`, `icons/`, `decorations/`）
- LLMがJSONで `elements`（パーツ配置）と `connections`（接続関係）を分離して記述
- コンポジションエンジン（~200行Python）が結線座標を算術的に計算し、確定的にSVGを合成
- パーツライブラリは拡張可能な設計（manifest.jsonにパーツ追加のみでコード変更不要）

## Impact

- Affected specs: `llm`, `ui`, `export`
- Affected code:
  - `llm_client.py` — 生成パイプラインの大幅拡張（コード生成、用途分類、ビジョンフィードバック）
  - `app.py` — UI拡張（用途選択、品質モード選択、フィードバック進捗表示）
  - `pptx_exporter.py` — 変更なし（SVG→PPTX変換は既存のまま）
  - 新規: `code_executor.py` — サンドボックスでのPythonコード実行
  - 新規: `prompt_templates/` — 用途別プロンプトテンプレート
  - 新規: `component_library/` — SVGパーツライブラリ + コンポジションエンジン
- New dependencies: `drawsvg` (MIT), `RestrictedPython` or subprocess sandbox
- **BREAKING**: なし（既存の直接SVG生成は「クラシックモード」として維持）
